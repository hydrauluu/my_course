## Что вы умеете после этого курса

Пройдём по тому что было. Не как список тем — а как набор навыков.

**Вы умеете читать исходный код.** Открыть Django, Pydantic, Toga — найти нужный класс, проследить цепочку вызовов, понять что происходит. Это редкий навык. Большинство разработчиков читают только документацию и Stack Overflow.

**Вы понимаете модель данных Python.** Объект, ссылка, тип, протокол — не как абстрактные слова, а как конкретный механизм. Когда что-то идёт не так, вы знаете где искать.

**Вы умеете получать исключения и читать их.** Traceback — это не страшно, это информация. Вы можете воспроизвести ошибку, изолировать причину, починить.

**Вы понимаете как Python устроен изнутри** — от подсчёта ссылок до метаклассов. Это даёт уверенность когда сталкиваетесь с незнакомым кодом.

## Как читать незнакомый репозиторий

Это навык который нарабатывается практикой. Вот рабочий процесс.

### Шаг 1 — Ориентация

Прежде чем читать код — прочитайте мета-документы:

- `README.md` — что это, зачем, как установить
- `CONTRIBUTING.md` — как устроена разработка, как запустить тесты
- `CHANGELOG.md` или `HISTORY.md` — что менялось и почему
- Структура директорий — где что лежит

```
django/
├── django/
│   ├── db/          # всё про базы данных
│   ├── http/        # запросы и ответы
│   ├── template/    # шаблонизатор
│   └── ...
├── tests/           # тесты — часто лучшая документация
├── docs/
└── setup.cfg
```

### Шаг 2 — Найти точку входа

Не читайте всё подряд. Найдите конкретную вещь которую хотите понять:

```python
# хочу понять как работает filter()
Book.objects.filter(author="Tolstoy")
```

Идём в исходник: `django/db/models/query.py` → метод `filter` → `_filter_or_exclude` → `Q` объекты → `add_q` → SQL компилятор. Каждый шаг — один файл, один метод.

### Шаг 3 — Читать тесты

Тесты — лучшая документация реального поведения. Они показывают:

- Как именно использовать API
- Какие edge cases существуют
- Что считается ошибкой, а что нет

```python
# tests/test_queryset.py в Django
def test_filter_by_field(self):
    Author.objects.create(name="Tolstoy")
    result = Author.objects.filter(name="Tolstoy")
    self.assertEqual(result.count(), 1)
```

### Шаг 4 — Запустить и сломать

Склонируйте репозиторий. Установите зависимости. Запустите тесты. Потом:

- Добавьте `print()` или `breakpoint()` в место которое хотите понять
- Запустите конкретный тест: `pytest tests/test_queryset.py::test_filter_by_field`
- Смотрите что происходит

`breakpoint()` — встроенный отладчик Python (pdb):

```python
def filter(self, *args, **kwargs):
    breakpoint()   # выполнение остановится здесь
    return self._filter_or_exclude(False, args, kwargs)
```

В pdb: `n` — следующая строка, `s` — шаг внутрь функции, `c` — продолжить, `p expr` — вывести значение выражения, `l` — показать код вокруг.

### Шаг 5 — Описать словами

Прочитали функцию — опишите что она делает одним абзацем. Если не можете — значит не поняли. Это не упражнение для новичков — это то что делают опытные инженеры при code review.

## Как контрибьютить в open-source

Контрибьютить — не значит сразу писать сложный код. Есть много способов участвовать.

### Что можно делать

**Документация** — самый недооценённый вклад. Найти неточность, добавить пример, исправить опечатку. По данным исследований, около 28% casual contributions в open-source — это документация: исправления опечаток, форматирование, переводы. При этом мейнтейнеры ценят это не меньше кода.

**Воспроизвести баг** — если кто-то открыл issue с багом, попробуйте воспроизвести и оставить комментарий с минимальным примером. Это экономит мейнтейнерам время.

**Написать тест** — найти код без теста, написать тест который проверяет поведение. Хороший способ разобраться в коде и сделать полезный вклад.

**Исправить баг** — взять issue с меткой `good first issue` или `help wanted`.

**Добавить фичу** — только после обсуждения в issue. Никогда не делайте большой PR без предварительного разговора.

### Рабочий процесс

```bash
# 1. Fork репозитория на GitHub (кнопка Fork)

# 2. Склонировать свой fork
git clone https://github.com/ВАШ_USERNAME/django.git
cd django

# 3. Добавить upstream (оригинальный репозиторий)
git remote add upstream https://github.com/django/django.git

# 4. Создать ветку для вашего изменения
git checkout -b fix/queryset-filter-bug

# 5. Сделать изменения, написать тест

# 6. Запустить тесты
python -m pytest tests/

# 7. Закоммитить
git add .
git commit -m "Fix: queryset filter incorrectly handles None values"

# 8. Запушить в свой fork
git push origin fix/queryset-filter-bug

# 9. Открыть Pull Request на GitHub
```

### Как писать хороший PR

- **Маленький и сфокусированный** — одна проблема, одно решение. Большие PR сложно ревьюить и они часто зависают
- **Описание** — что изменено, почему, как проверить. Ссылка на issue
- **Тесты** — каждое изменение поведения должно быть покрыто тестом
- **Стиль** — следуйте стилю проекта, не переформатируйте чужой код без необходимости

```markdown
## Что изменено
Исправлена обработка None значений в `QuerySet.filter()`.
Раньше `filter(field=None)` генерировал некорректный SQL.

## Почему
Closes #12345

## Как проверить
```python
# до фикса — падало с SQL error
Book.objects.filter(author=None)

# после — возвращает книги без автора
```

## Тесты

Добавлен `tests/test_queryset.py::test_filter_by_none`

```

### Что делать после review

Мейнтейнер оставил комментарии — это нормально. Это не значит что ваш код плохой. Ответьте на каждый комментарий: либо исправьте, либо объясните почему не согласны. После исправлений — попросите повторный review.

## Как строить свою систему работы с кодом

### Читайте PEP-ы

Python Enchancement Proposals — это не сухие спецификации. Это документы где объясняется **почему** язык устроен именно так. Самые важные:

- [PEP 8](https://peps.python.org/pep-0008/) — стиль кода. Не потому что так принято, а потому что понимаешь логику
- [PEP 20](https://peps.python.org/pep-0020/) — The Zen of Python. `import this`
- [PEP 484](https://peps.python.org/pep-0484/) — типизация, мы разбирали
- [PEP 492](https://peps.python.org/pep-0492/) — корутины, мы разбирали
- [PEP 572](https://peps.python.org/pep-0572/) — walrus operator `:=`, интересная история принятия

### Читайте CPython

Интерпретатор Python написан на C и Python. Многое можно читать прямо на GitHub:

- [Lib/](https://github.com/python/cpython/tree/main/Lib) — стандартная библиотека на Python. `functools.py`, `collections/__init__.py`, `pathlib.py` — всё открыто
- [Objects/](https://github.com/python/cpython/tree/main/Objects) — реализация встроенных типов на C

Начните с `Lib/functools.py` — там `lru_cache`, `wraps`, `partial`. Понятный Python-код с подробными комментариями.

### Инструменты которые стоит знать

**Отладка:**
```python
breakpoint()            # встроенный pdb
import pdb; pdb.set_trace()   # то же самое, явно
```

**Интроспекция:**

```python
import inspect

inspect.getsource(func)        # исходный код функции
inspect.signature(func)        # сигнатура
inspect.getmembers(obj)        # все атрибуты и методы
inspect.getmro(cls)            # MRO класса
```

**Профилирование:**

```python
import cProfile
cProfile.run("your_function()")

# или через командную строку
python -m cProfile -s cumulative script.py
```

**Память:**

```python
import tracemalloc

tracemalloc.start()
# ... код ...
snapshot = tracemalloc.take_snapshot()
top = snapshot.statistics("lineno")
for stat in top[:10]:
    print(stat)
```

## Финальный разговор

Этот курс давал вам не Python. Python — просто язык на котором мы смотрели идеи.

Настоящие вещи которые вы взяли:

**Читать документацию как первоисточник.** Не Stack Overflow, не туториалы — а то что написали авторы языка. Это навык который работает для любого языка и любой библиотеки.

**Получать исключения и не бояться.** Ошибка — это информация. Чем больше вы ошибаетесь в контролируемой среде, тем меньше ошибок в продакшне.

**Описывать код словами.** Если не можете объяснить что делает функция — значит не поняли. Это и есть инженерное мышление: понять систему достаточно хорошо чтобы объяснить её другому человеку.

**Читать чужой код без страха.** Любой код написан людьми. В нём есть логика, компромиссы, история. Всё это можно понять.

Дальше — практика. Возьмите реальный проект, библиотеку которой пользуетесь, и начните читать исходник. Это лучшее что можно сделать после окончания курса.
